Why & what's changing

3 min

Today every customer's archive bucket trusts the same shared Coralogix role. Shauli flagged this as a security risk — the role ARN isn't a secret, so a hostile actor could ask Coralogix to act against another tenant's bucket (the AWS "confused deputy" pattern). This change closes that gap by giving each customer their own IAM role gated by an External ID.

The change in one line: Two new fields appear in the Setup Archive Physical Location form. Customer does the AWS-side work first, then enters two values into our UI.
New field 1
Role ARN
The ARN of the IAM role the customer created in their AWS account. We use it on every AssumeRole call when writing to their bucket.
New field 2
External ID
A free-text string the customer invents and places in both their IAM trust policy and our form. Not Coralogix-issued. Acts as a per-customer secret on every AssumeRole.
Security parity, not differentiation. No marketing story here — this is an AWS best-practice closeout. Customer obligation goes up (they own role + bucket policy), which is the CS impact surface.

The flow — three lanes

5 min

AWS work happens first. Coralogix UI is the final step — we just store and use values the customer gives us. Every error edge loops back to AWS, because the source of truth lives there.

3

New customer vs. existing customer

4 min

The new-customer path is trivial. The migration story is the one CS needs to internalize — because the headline ticket class will be "will switching break my archive?"

THE message for CS: Migration is additive. Customer adds the new role alongside the default Coralogix role. Nothing breaks. They remove the default on their own timeline. No forced cutover. No data-loss window.

New customer new auth method only

New customers never touch the old shared default role. They start directly on the new auth method.

Single bucket policy statement: AllowCustomerArchiveRole. Create bucket → create role with External ID in trust policy → enter Role ARN + External ID in Setup Archive → done.

Bucket policy — single AllowCustomerArchiveRole statement, no default role

Existing customer additive migration

Bucket policy carries both statements simultaneously: the old default role and the new customer role.

Compaction continues writing via either path. Customer removes the default statement on their own timeline once they've validated the new role.

Bucket policy — AllowCoralogixDefaultRole + AllowCustomerArchiveRole coexist during migration only
The new primitive both customer types configure — the IAM role's trust policy with sts:AssumeRole + StringEquals on sts:ExternalId. This is the line that gates the whole flow.
Volume context: 393 customers across all regions have >1 physical location on default/logs. 90% have 2 locations; long tail of 4 customers with 4–6 locations. AP1 + EU2 + CX820 = 75% of volume. We'll need a bucket-inventory UI surface — customers can't track this themselves.
