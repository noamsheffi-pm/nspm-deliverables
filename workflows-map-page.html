📄 Workplan Maintenance Loop

The state-keeping mechanism for 00-workplan.md. Every initiative has one. Three writers + one reader keep it current. Each owns specific sections.
Sources Writers (manual) Workplan sections owned 💬 Session work ─┐ ┌─ Where I Left Off ├─→ /session-close ├─ Session Log 📝 New meeting notes ─┘ (per session, ├─ Decisions ledger (since last-reviewed) end of work) ├─ Open Items / Resolved ├─ Watch List ├─ Blockers └─ Lifecycle phase tables 🎫 Jira (parent only) ─┐ ┌─ Tickets table (single row) ├─→ /roadmap-review ├─ frontmatter status 📋 Notion roadmap ─┘ (weekly portfolio) ├─ frontmatter urgency Step 5c + 5c.1 └─ frontmatter last-reviewed (deep walk for opt-in workplans with ## Tickets) 🎫 Jira (full hierarchy: ─→ /initiative-review {KEY} ┌─ Tickets table (full reconciled) Initiative→Feature (on-demand per initiative) ├─ Key Files: tracker back-link →Epic→Task) Step 5b cross-link ├─ frontmatter status (advance only) └─ frontmatter last-reviewed 📄 00-workplan.md ─→ /today ─→ ⚠️ Workplan Drift section in dashboard (daily, read-only) (nudges user to run a writer)
